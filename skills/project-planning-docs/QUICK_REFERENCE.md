# Project Planning Docs — Quick Reference

A one-page cheat sheet. See [SKILL.md](SKILL.md) for the authoritative instructions.

## The flow

| Step | What happens |
|------|--------------|
| 1. Project type | Determine **software** or **game**, then read the matching reference file. |
| 2. Intake | One free-text description + 1–2 rounds of ≤3 tappable questions. |
| 3. Draft | Produce every document in the suite as numbered `.md` files. |
| 4. Present | Show the files (vision doc first); offer a Word (`.docx`) export. |

## Intake questions

**Both tracks:** target platform · team size · timeline horizon · development methodology

**Software round 1:** product type · target platform · team size
**Software round 2:** timeline · methodology · build approach (MVP vs full)

**Game round 1:** genre · platform · player mode
**Game round 2:** art style · monetization · team size (plus timeline & methodology)

## Document suites

**Software:** `01` Vision/Concept · `02` PRD · `03` Technical Design · `04` Project Plan · `05` User Stories/Backlog · `06` Test/QA Plan · `07` Risk Register

**Game:** `01` Concept/Pitch · `02` GDD · `03` Technical Design · `04` Art Bible · `05` Project Plan · `06` Test/QA Plan · `07` Risk Register

## Rules of thumb

- **Specific over generic** — draft real feature names, user types, and schedules from the intake.
- **`[TODO]` for unknowns** — never invent budgets, dates, or legal commitments.
- **Visuals** — describe wireframes/concept art and leave a placeholder; don't fake them in text.
- **Consistency** — same project name, audience, and scope across every document.
- **"Assume defaults"** — if the user says so, stop asking and proceed, stating your assumptions.
- **Methodology unsure?** — default to Agile/Scrum; reserve Waterfall for fixed-scope, fixed-deadline work.
- **Save real files** — the numbered `.md` files are the deliverable, not the chat.
